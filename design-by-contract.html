Design by Contract

Clear agreements that make software modules reliable

Why Contracts?

What a Contract Does

A contract sets out rights, responsibilities, and what happens if either side breaks the agreement.

Employment Example

You agree to work certain hours and follow rules. The company pays you a salary and benefits. Both sides meet their obligations — everyone benefits.

People have used contracts for thousands of years to interact fairly. Can we use the same idea so software modules interact fairly? The answer is yes.

Why It Matters

Prevents Wrong Assumptions

Clear contracts make requirements explicit — no guesswork about what a function expects or returns.

Reduces Runtime Bugs

Invalid inputs are caught early before they cause subtle, hard-to-debug issues later.

Improves Reliability

Contracts verify correctness at runtime and fail with clear error messages.

Easier Debugging

When something fails, contracts show exactly what failed and why.

What Is Design by Contract?

Design by Contract (DBC) is a way to document and check what your functions and classes must do and must not do.

Contracts can be enforced with typed parameters, exceptions, and documentation — making rules explicit in your code.

The examples in this presentation use PHP; the same ideas apply to any language.

What is a correct program? One that does no more and no less than it claims to do.

Documenting and checking that claim is the heart of DBC.

Three Parts of Every Contract

Precondition

What must be true before the function is called — the function's requirements.

Don't call a function unless its preconditions are already met. It is the caller's job to pass good data.

Postcondition

What the function guarantees after it finishes — the state when it is done.

A postcondition means the function will finish — infinite loops are not allowed.

Invariant

A condition that is always true from the caller's point of view.

It may be broken briefly inside the function, but it must be true again before control returns to the caller.

The Deal

If all preconditions are met by the caller, the function guarantees that all postconditions and invariants will be true when it completes.

If the Contract Breaks

A remedy is invoked — maybe an exception is raised, or the program stops.

Breaking a contract is a bug. It should never happen in normal use.

Not for User Input

Preconditions are not for checking user input on a form or console.

User validation belongs in the controller or form layer — before your function is ever called.

Caller vs Function Responsibility

The Caller

  • Provide valid inputs
  • Meet all preconditions
  • Respect parameter constraints
  • Handle documented exceptions

Mutual Agreement

Both sides work together for reliable, predictable behavior — code that is easier to understand, test, and maintain.

The Function

  • Guarantee correct output
  • Meet all postconditions
  • Maintain invariants
  • Document behavior in PHPDoc

Invariants vs Assertions

PHP examples below — same ideas apply in any language

Invariants

What: Rules that must stay true from the caller's view — before and after every call.

Scope: Object state or data passed between functions.

Purpose: Protect consistency over time, not just one input check.

// PHP example — class invariant private float $balance = 0; public function withdraw(float $amount): void { $this->balance -= $amount; assert($this->balance >= 0, 'Balance invariant'); }
vs

Assertions

What: Runtime checks that a specific condition is true at a point in code.

Scope: One moment — entry, exit, or mid-function.

Purpose: Catch contract breaks immediately (dev, test, or prod).

// PHP examples — assertion variants assert($amount > 0); // basic assert($x >= 0, 'Non-negative'); // with message if ($amount <= 0) throw new ...; // exception (prod) function f(int $n): int { ... } // type hint

In short: An invariant is the ongoing rule your module must obey. An assertion is how you check preconditions, postconditions, or invariants at runtime. Assertions can be disabled in production; invariants should still be enforced via exceptions where it matters.

Example: PHP Deposit

Banking deposit — preconditions and postconditions

// PRE + POST contract checks function acceptDeposit(int $accountId, float $amount): int { // PRE: amount must be positive if ($amount <= 0) { throw new InvalidArgumentException('Amount must be greater than zero.'); } // PRE: account must be open if (!accountIsOpen($accountId)) { throw new RuntimeException('Account is not open.'); } $transactionId = createTransaction($accountId, 'deposit', $amount); // POST: transaction must be recorded if (!transactionExistsOnAccount($accountId, $transactionId)) { throw new LogicException('Deposit was not recorded.'); } return $transactionId; }

Preconditions

Positive amount. Account must be open.

Postcondition

Transaction id exists on the account.

On failure

Exception thrown before bad state is saved.

Example: PHP Withdraw

Bank account withdrawal — preconditions, postcondition, and invariant

// PRE + POST contract checks (not user-form validation) function withdraw(int $balance, int $amount): int { if ($amount <= 0) { throw new InvalidArgumentException('Amount must be greater than zero.'); } if ($amount > $balance) { throw new RuntimeException('Insufficient balance.'); } $newBalance = $balance - $amount; // POSTCONDITION: output guarantee if ($newBalance < 0) { throw new LogicException('Balance cannot be negative.'); } return $newBalance; }

Preconditions

Positive amount and sufficient balance.

Postcondition

Non-negative final balance.

Invariant

Balance never goes below zero.

What This Example Shows

Lessons from the withdraw() function above

Preconditions Protect Inputs

Validates $amount is positive and balance is sufficient before processing. No unexpected inputs break the logic.

Postconditions Protect Outputs

Guarantees balance never goes negative before returning. Caller receives exactly what was promised.

Exceptions Are Production-Safe

Typed exceptions with clear messages — safe failure without silent corruption.

Contracts Enable Predictability

Self-documenting, testable behavior — no hidden requirements or surprises.

Semantic Invariants

Inviolate requirements — a "philosophical contract"

Debit Card Switch

A user must never have the same transaction applied to their account twice.

On failure, err on the side of not processing rather than processing a duplicate.

Laws vs Policies

Semantic invariants are fixed laws central to the meaning of a system — not policies that change with management.

"Err in favor of the consumer."

A clear, unambiguous contract with all users of the system. Document it — in a requirements doc or on the team whiteboard.

Best Practices

01 — Keep Contracts Simple

Focus on essential validations only. Validate critical business rules and safety-critical conditions. Avoid trivial checks.

02 — Fail Fast with Clear Errors

Throw exceptions immediately when contracts are violated. Use descriptive messages and appropriate exception types.

03 — Document Expectations

Use PHPDoc for parameters. Document preconditions and postconditions. Provide usage examples.

04 — Use Exceptions for Safety

Prefer exceptions over error codes — harder to ignore, better context. Don't swallow exceptions silently.

Pro tip: Start with preconditions and postconditions. Add invariants as you get comfortable.
